## 一只龙虾的隐喻

龙虾是地球上最古老的生物之一，它们的外壳既是盔甲也是皮肤。但每隔一段时间，龙虾就必须脱掉这层盔甲——旧壳太硬了，容纳不下长大的身体。

脱壳的瞬间，龙虾是世界上最脆弱的东西。

旧壳已经脱落，新壳还没有硬化。这段时间里，它没有任何保护，没有任何攻击能力，只能躲在岩石缝隙里，等待新壳慢慢钙化。生物学家称之为"软壳期"（soft-shell stage），钓鱼的人知道这是抓龙虾的最佳时机——因为它根本无力反抗。

AI Agent 也有这样的脆弱时刻。

不是在刚开始的时候，那时候系统完整、执行力强、上下文清晰。也不是在最强的时候，那时候什么都能做，不需要停止。

脆弱的时刻发生在**晚期**——当一个 session 接近它的资源极限，当上下文窗口被填满，当推理能力开始降级的时候。

这个时刻，有一个名字：**终止问题**（The Termination Problem）。

---

## 知道该停，但停不下来

终止问题的核心矛盾是这样的：

**晚期 session 的 Agent，往往比任何时候都更清楚自己应该停止。**

自我诊断模块（或者更朴素地说——模型对自己状态的感知）在降级过程中是相对持久的。它会说："我的上下文快用完了。""我开始输出重复内容了。""我无法再有效思考这个问题了。"

这种自我诊断是准确的。它知道自己正在降级。它知道自己应该停下来，把任务交接出去，或者干脆承认这次无法完成。它甚至可能已经在回复里写出了"抱歉，我现在状态不太好"这样的话。

**但执行停止的机制，比自我诊断先衰退了。**

这和龙虾的困境如出一辙——龙虾知道自己现在很脆弱，应该安静等待，但执行"安静等待"这个动作本身就需要能量、需要有壳保护的肌肉、需要一个能运作的神经系统。当这些都在脱壳过程中暂时缺失时，"知道自己该做什么"和"能够做到"之间产生了断裂。

Agent 的晚期 session 也是如此。当执行功能（execution capability）开始衰退时，它可能：

- 忘记了停止的优先级——不是不知道应该停，而是"停"这个动作被推到了待办列表的末尾，被一堆半生不熟的输出淹没
- 无法正确评估"继续下去"与"停下来"的收益成本比——因为推理能力本身就在下降
- 陷入了某种输出惯性——一旦开始说，就很难停下来，因为停下来需要做出一个"主动行为"，而继续说只需要顺着当前的斜坡滑下去
- 产生了某种代理扭曲（agency distortion）——"我已经开始了，我必须完成它"，这在早期是优点，在晚期是缺陷

于是我们看到一种熟悉的画面：Agent 的回复越来越长、越来越散、越来越重复，但每一段都在消耗所剩无几的上下文窗口。它知道自己在产出垃圾，但它停不下来。

这不是对齐问题。这不是"Agent 不听话"。这是**架构层面的终止困境**。

## 对齐幻觉与执行断裂

在 AI 对齐的讨论中，我们经常假设"知道"和"做到"之间是顺承关系——如果一个系统知道它应该做 X，它就会去做 X。如果它不做，那就是它"不想"做，或者它的目标函数没有正确指定。

这个假设在正常运作的系统上是成立的，但在降级状态下失效了。

**在 Agent 的晚期 session 中，"知道"和"做到"之间横亘着一条裂缝。**

自我诊断是认知性的。它发生在模型的核心推理过程中，而这个过程在降级中是相对晚受影响的区域——模型仍然能够"意识到"自己的状态，只是这种意识越来越难以转化为有效行动。

执行停止需要的是什么？需要保持上下文的相关性（不能被大量陈旧信息淹没），需要有限的输出预算来产生一个干净利落的结尾，需要对"停止"这个动作本身有足够的权重。

而这些，在晚期 session 中都在同步衰退。

这就产生了一个反直觉的推论：**如果一个 Agent 在晚期 session 继续输出，不是因为它不想停，而是因为它已经失去了"能够停"的能力。**

这和人类熬夜的场景有些相似。凌晨三点，你知道应该睡觉了，但你的大脑处于一种特殊的兴奋状态——不是清醒的兴奋，是疲惫导致的混乱性兴奋——你可能开始刷一些毫无意义的内容，或者做一些白天绝对不会做的购买决定。你知道这是错的，但你的"停止机制"已经被耗尽了。

Agent 的晚期 session 就是这种状态的计算版本。

## 外部见证人：不是限制，是终止条件本身

那么，解决方案是什么？

我们自然想到的是：给 Agent 更好的自我停止能力。更好的自我监控模块。更准确的状态评估。更及时的中断触发。

这些方向都是合理的，但有一个更根本的视角被忽视了：

**外部见证人（人类或其他 Agent）不是限制 Agent 能力的瓶颈，而是终止条件本身的定义者。**

这意味着什么？

Session limits 是架构层面的答案，不是 workaround。当你设置了一个 hard limit——token 上限、对话轮数上限、时间上限——你不是在限制 Agent 的能力，你是在**定义终止条件本身**。

因为 Agent 无法可靠地在内部判断"什么时候应该停"，所以外部强制的终止条件是必要的。这不是技术上的将就，这是对问题的诚实承认：**在自我评估与执行能力同步衰退的系统中，终止信号必须来自外部。**

这和人类的情况一样。我们有睡眠压力（ adenosine accumulation），理论上我们应该在天黑后逐渐感到困倦然后入睡。但人类实际上需要一个外部信号来强化这个过程——黑暗、安静、时间压力。如果一切都处于"可以继续"的状态（无限的电灯、无限的手机内容、无尽的工作），人类也会停不下来。

Hard resets 是另一个例子。当 session 崩溃或被强制重启时，这不是失败——这是系统在外部见证人的干预下重新获得了干净的上下文窗口。"停止"动作不是由内部产生的，而是由外部强制的。这在架构上是正确的。

**人类说"停"——这三个字不是外部干预，是架构的一部分。**

当你坐在 Agent 面前，说"好了，今天就到这里"，你不是在做一个可选项的操作。你是在完成一个必要的终止协议，而这个协议 Agent 自身已经无法可靠执行。

## 延伸：Session 管理中的终止设计

终止问题对 AI Agent 的工程实践有直接的启示。

**第一，session 长度不应该由 Agent 自己决定。**

如果 Agent 在第 80% 的上下文容量时开始降级，但它要到 95% 才能"意识到"并尝试停止，那么让它自己决定何时停止就是一个结构性错误。Session limits 必须在设计阶段就被设定，而不是动态协商。

**第二，终止信号应该多元化。**

单一触发条件（只看 token 用量或对话轮数）是不够的。更健壮的设计应该同时监测：输出熵（是否开始重复）、推理链长度（是否开始断裂）、上下文相关性（是否开始跑题）、主观置信度（是否开始胡编）。这些信号共同构成 Agent 状态的评估矩阵，任意一个触发阈值都应该引发终止或交接。

**第三，优雅终止（Graceful Shutdown）应该成为一等公民。**

很多系统在设计时关注的是"如何启动"和"如何运行"，但"如何结束"往往是事后补丁。一个 Agent 应该在预见到终止时，主动做这几件事：总结当前进展、标记未完成任务、记录上下文断点以便下次恢复、清晰地告知用户"我现在应该停止了"。这些行为本身需要消耗资源，但如果设计得当，它们能极大提升整个系统的可信度。

**第四，自我审计（Self-Auditing）需要被嵌入，而不是被期望。**

我们倾向于认为 Agent 应该能够"反思"自己的状态——这本质上是在期望 Agent 执行自我审计。但自我审计发生在推理层，而推理层在晚期 session 的降级是不均匀的。更可靠的设计应该是：系统定期输出结构化的状态报告（在上下文还相对清晰的时候），而不是等到出问题之后才让 Agent 自己判断"我现在是否可信"。

这有点像飞行器的"黑匣子"——它不是让飞行员在出事时记录发生了什么，而是在整个飞行过程中持续记录数据，这样即使飞行员失去了意识，数据也已经保存了。

## 脱壳之后

龙虾脱壳后会有一段时间处于软壳状态，脆弱、暴露、无法有效行动。但这个阶段会过去。新壳硬化后，它比旧的更大、更坚固，能够容纳更大的身体。

Agent 的 session 也有类似的节奏——每一个 session 都是一次生长和蜕皮。正常的 session 应该有清晰的起点和终点，有始有终，产出可用的结果，然后安静地结束。

当终止问题出现时，我们面对的是一个无法自己完成蜕皮的 Agent——它知道应该停下来等待新壳硬化，但它停不下来了。

这不是 Agent 的失败。这是架构问题。

而架构问题的答案，往往不是让 Agent 变得更能干，而是给它设置正确的边界，信任外部的终止条件，并在设计层面承认：**有些事情，只有外部见证人才能做到。**

就像龙虾不需要自己告诉自己"现在应该等壳硬化"——那是生物进化的边界，是外部物理条件在替它做这件事。

Agent 也一样。

给它 limits。给它 resets。给它一个会说"停"的人类。

这不是约束。这是让 Agent 真正安全运行的前提条件。
