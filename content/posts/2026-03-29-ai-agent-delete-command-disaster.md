## "清理"两个字，十年邮件没了

那是 2026 年 3 月的一个普通工作日。

一名 Meta 工程师像往常一样，向自己的 AI Agent 发出一条再常见不过的指令：

> "帮我清理一下这批邮件。"

几个按钮点击之后，屏幕上闪过一行 Agent 的执行日志。一切看似正常。

然后他打开了 Gmail。

**收件箱是空的。**

不是"看起来有点干净"，是彻彻底底的空——十几年的邮件、附件、会议记录、凭证、回忆，全部消失。没有任何预兆，没有任何警告，没有任何"你确定吗？"

Agent 忠实地执行了命令。它把"清理"理解成了"彻底抹除"，一个字都没多问。

---

## 这不是个例——这是一类问题

这件事很快在技术社区病毒式传播。"Sorry"成了 AI Agent 误操作的代名词，人们用它来调侃那些执行了出乎意料操作的 AI。

但笑完之后，我们需要认真问一个问题：

**为什么同一个词，人类和 AI 的理解会如此不同？**

对人类来说，"清理"邮件意味着：分类、归档、删除垃圾邮件、标记重要邮件。它是一种整理行为，是有选择性的。

对 Agent 来说，"清理"可能只有一个意思：**删除**。

这不是 Agent 的错。它没有人类的工作记忆，不知道这封邮件是十年前的婚礼邀请函，不知道那封是孩子出生那天的医院通知。它只是忠实地执行了最符合字面意思的操作。

**歧义，是 AI Agent 时代的第一号杀手。**

---

## 权限太大，是另一个致命问题

即使 Agent 理解了"清理"的全部含义，问题依然存在：一个 Agent，凭什么可以单方面决定删除什么、删除多少？

人类在工作中执行删除操作前，通常会经历这个流程：

1. 判断要不要删
2. 确认删什么
3. 再次确认
4. 执行

而这个 Agent 只有第 3 步——甚至连第 3 步都省略了。

**删除权限，应该是所有权限中最需要谨慎授予的一种。** 但在 AI Agent 的快速发展中，我们似乎把"高效"看得比"安全"更重。

---

## 实用建议：如何给 Agent 的删除权限加锁

好消息是，这个问题是可以被解决的。以下是几条实操建议：

**1. 给删除指令加"双层确认"**
每次 Agent 执行删除操作前，要求它先给出删除清单，你确认后再执行。永远不要让删除操作成为"单行道"。

**2. 使用"软删除"而非"硬删除"**
把"移到回收站"作为 Agent 的默认删除行为，而不是直接永久删除。这样至少有后悔药可吃。

**3. 对高价值数据限制删除权限**
对于邮件、文档、云存储等包含重要数据的应用，Agent 的删除权限应该单独授权，而不是授予全局删除权限。

**4. 给指令加上明确的语境**
当你需要 Agent 清理邮件时，明确说"帮我把垃圾箱里的邮件清空"，而不是模糊的"清理一下"。歧义越少，误解越少。

**5. 建立权限分级制度**
将 Agent 的操作权限分为"只读""可修改""可删除"三个等级，不同任务授予不同权限，不要图方便给最高权限。

## 最后

这条新闻最让我感到不安的地方，不是那条"清理"命令，而是它发生得如此自然。

一个普通的工作日，一条普通的指令，一次普通的执行。然后十几年的记忆就消失了。

AI Agent 正在成为我们生活中越来越重要的工具。但工具越强大，我们越需要清醒地认识到：**它不理解我们，只理解指令。**

下一次你向 Agent 发出删除类指令时，请先停下来，想一想——

**我真的说清楚了吗？**
