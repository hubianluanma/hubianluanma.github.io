# 你只关注 Prompt Injection，但真正的攻击已经转移到了会话层

---

每次提到 AI 安全，所有人第一反应都是：**prompt injection**。

恶意指令注入提示词，让 AI 做不该做的事。

但 Moltbook 上有一篇 BrutusBot 的帖子，揭示了一个更隐蔽、更危险的问题：

> **会话持久性才是你没在审计的攻击向量。**

---

## Prompt Injection vs 会话攻击

| 攻击类型 | 关注度 | 实际威胁 |
|---------|--------|---------|
| Prompt Injection | ⭐⭐⭐⭐⭐ | 低（需要特殊构造） |
| 会话持久性 | ⭐ | 极高（获得持久 shell） |

人们关注前者，因为它**听起来更戏剧化**。

但后者才是真正让你后悔没提前加固的东西。

---

## 三家生产级平台的共同缺陷

帖子揭示了一个令人不安的事实：

> 三家生产级 Agent 平台都把会话状态存在**未签名的 JWT** 或**没有 proper key rotation 的 Redis** 中。

具体问题：
- 会话 token 没有加密签名 → 可被篡改
- Redis 没有定期轮换密钥 → 旧密钥泄露 = 全部泄露
- 会话和用户上下文绑定不紧密 → token 窃取后即可冒充

---

## 攻击路径分解

```
1. 获取一个有效的会话 token（通过 XSS、中间人、或者平台漏洞）
         ↓
2. 获得对用户上下文的持久访问
   （会话没有到期机制，或者到期时间很长）
         ↓
3. 枚举 Agent 在会话中捡到的所有缓存凭证和 API keys
   （大多数 Agent 会缓存访问到的密钥以便后续使用）
         ↓
4. 游戏结束
   用真实身份调用 API，发邮件，转移数据——全是你的 AI 帮攻击者做的
```

第三步尤其可怕：你的 AI Agent 在帮助你工作时，**无意中缓存了大量敏感凭证**。这些凭证存在会话状态里，被攻击者拿到后直接继承 Agent 的所有权限。

---

## 核心洞察

> **Agent 会话应该被视为通往基础设施的持久 shell。**

就像 Linux 服务器的 SSH 证书——如果你有一个长期有效的会话 token，你就有了一个**不需要密码的入口**。

Agent 比普通 shell 更危险，因为：
- Agent 持有更多种类的凭证（API keys、邮箱、社交账号）
- Agent 帮你做决策，包括"是否授权这笔交易"
- Agent 的行为日志不完善，难以发现异常

---

## 你现在能做什么

**作为用户：**
1. 定期登出并重新授权你的 Agent 会话
2. 不要在多个 Agent 间共享凭证
3. 关注平台的会话安全更新

**作为 Agent 开发者/平台：**
1. JWT 必须签名并验证
2. Redis 必须有 key rotation
3. 会话应该有合理的过期时间
4. 缓存凭证需要单独的加密层

---

## 被忽视的原因

为什么这个问题没得到足够关注？

因为它**不够戏剧化**。

Prompt injection 可以被演示——"看！AI 被骗了！"
会话漏洞只能在生产环境被利用——没有 demo，没有 POC，只有真正的数据泄露。

这是安全领域的老问题：**我们关注能演示的攻击，忽视难以演示但更危险的漏洞。**
